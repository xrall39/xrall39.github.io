Java源码之HashMap（JDK1.7）

发表于 2019-06-05 | 更新于 2019-07-25 | 分类于 Java ， 源码

该文章记录学习源码时候的一些问题和看法，当然下面文章就要通过我手来进行解读分析。本文章只适用于1.8以下版本，我看源码的时候用的1.7。

前言吐槽

在我独自一人看源码的时候发现一些问题，当然都是针对我个人的。

首先自己看时候发现英语单词看不明白，导致成员变量的含义不清楚，就无法顺利读懂源码。

其次就是，学习源码最好是带着疑问一口气看完，然后在看是否能回答上那些问题，至于哪里有问题，可以百度面试就会出现一堆HashMap的问题。

最后用自己双手来实践一个简单的拥有存储，读取的HashMap。

分析前的准备

分析问题之前我们最好要利用你已知的内容来进行推理。

数据结构

1.8之前采用数组+链表形式。

  • 数组：寻址容易，插入和删除困难；
  • 链表：寻址困难，插入和删除容易。

拉链法，我们可以理解为“链表的数组” ,HashMap在存储数据的时候就是用的拉链法来实现的。

数据储存介质

因为是K-V键值对形式储存，那么猜测储存的是一个Object（对象）

键值对如何选择数组中的位置

我们知道一点，HashMap是无序的，也就说我这个键值对是通过随机或者某种算法来选择数组中的位置。

既然是通过某方法来确定数组中的位置，那我们肯定得知道数组大小吧？不可能无限大吧？

假设初始数组大小是8那么我们数据存放的位置下标必然是0-7，这个假设没问题吧。

那我觉得最简单的可能就是一个数%8肯定就是0-7之间吧！至于一个数怎么演变来的，可以自己想一下，通过hashcode？还是说random？这里就不多猜了。

链表

既然我们都知道底层用了数组+链表的形式，那为何要用链表呢？在上面了解到他是随机分配，那么这样就有一个问题那就是可能出现两个数据占同一个坑的情况，那我们该如何处理？你不可能，后来者居上直接数据给覆盖了吧。所以这里就出现了链表，那问题就来了！我这个链表从哪里插入啊？从头吗？还是从尾？

准备总结

基本上我们，知道了用什么来实现(数组+链表)，数据是怎样一个东西(Node<k,v>)，如何选择位置(随机后取模)，然后选到同一个位置后链表的情况，当然这些都是粗糙的了解一下，下面就正式来查看源码，分析源码。

HashMap属性

Q：为什么这些数字非得用左移右移，非用1 << 4而不直接写16

A：因为你任何数字到底层都是二进制，所以1 << 4的效率比直接16快

PS：注意最后一个属性，他是扩容或者更改了结构才计数，如果添加元素和map里面有一个key相等是，只会替换掉原来的值，而并不会导致结构产生变化。

PS：里面有一个table它是通过EMPTY_TABLE类型强转过来的。

数据介质

添加方法

到这里要缓一缓数据量有点大~首先你在这里看到了我们准备是进行推测的东西，第一个是数据存储介质，第二个就是如何选取下标，第三个如果该位置有数据那链表怎么形成，头部还是尾部。这样一个完整的顺序就差不多学习完了。

Q：为什么生成一个数组容量非得要2的幂次方?

A：这里涉及到一个二进制的问题，把眼光放长远一点，结合下标的选取来进行解释，这里暂时先保留下问题。

Q：为什么要用一个hash方法来获取hash值，不是已经有了已经有了hashcode方法了吗？

A：这一切的一切都是为了减少冲突概率，让进来的值能更加平均的选取数组的下标。

Q：取下标当中采用的h & (length-1)这是一个什么操作？

A：推测的时候最简单可能就是h % 8(8是我们假设的容量)其实他这个写法翻译过来就是h % n(n为容量)，他这样书写比我们直接取模效率更快。

Q：既然你说取模为什么还要length-1呢？

A：这里我们就可以吧第一个问题拿出来，为什么数组长度非得2的幂次方(这也许就是大佬的伏笔吧)。

Q：为什么Key为null没报错啊？讲道理该报错啊？

A：是的，我以前也是以为插入null会出现空指针异常，然而他并没有。那你注意到了key=null插入的位置吗？

Q：如何采用的链表？为什么采用头部插入？

A：仔细查看Entry类你会发现一个他有一个next的属性，next属性指向下一个Entry，就串起来了，只不过他没有向上指向，所以他有了一个操作向下推动作，也就是table[bucketIndex]=new Entry。

总结

说实在，我看源码就发现一个最大问题就是，当我们跟进去后不知道他这个干啥用的，或者说变量名含义是什么。虽然他每一个都有注释，但是对于我来说还是很吃力的。比如HashMap的成员变量： load Factor直译过来加载因子，加载因子？然后就疑惑了，意思都看不懂看蛋蛋啊，更别提长篇大论代码了。说到底还是自己英语底子不行……所以我在分析开头把源代码中属性进行罗列并用最简洁的语言来进行解释。

我总结的顺序是根据平常写代码的逻辑来进行推导，记录书写的。首先new HashMap，然后添加键值对，然后获取对应的键值对。当然删除这些就没有记录，一方面是因为我当时网上学习的时候大家也没讲解过，二方面是，希望看到本文章的人也能学会自我推导，因为最终路都是要自己走的。
